
## 2025-02-28 - Fast API Event Loop Blocking with Synchronous Persistence Calls
**Learning:** In `backend/main.py`, the SQLite persistence calls (`create_transfer_record`, `list_transfers`, `get_transfer`, and `set_agent_b`) are executed synchronously within FastAPI's `async def` endpoints. Because these endpoints are defined as `async def`, FastAPI runs them on the main event loop rather than in a thread pool. Consequently, any blocking I/O operation like SQLite database access directly blocks the entire ASGI event loop, preventing concurrent request handling while the DB operation is active. While `asyncio.to_thread` is already correctly used for the heavy `pyttsx3` TTS generation, it is missing for the database persistence layer.
**Action:** All direct database interaction calls within `async def` functions in FastAPI must be wrapped with `await asyncio.to_thread(...)` to offload the blocking operations and allow FastAPI to continue serving other concurrent requests. Wait, `asyncio.to_thread` accepts `*args` and `**kwargs`, so we can pass arguments using `functools.partial` or directly with `*args` and `**kwargs` in python 3.9+. In python 3.9+ `asyncio.to_thread` passes kwargs directly to the target function, but actually wait, `asyncio.to_thread` takes `func, *args, **kwargs`.
